# 云效状态 ID 映射指南

本文档提供云效工作项状态 ID 的完整参考和动态查询方法。

## 状态阶段概览

云效使用三阶段状态模型：

| statusStage | 名称 | 说明 |
|-------------|------|------|
| 1 | 未开始 | Bug 已报告但工作未开始 |
| 2 | 进行中 | Bug 正在被处理 |
| 3 | 已关闭 | Bug 已解决并关闭 |

## 常见状态 ID

**⚠️ 重要**：状态 ID 可能因云效项目而异。生产环境务必使用动态查询。

### 阶段 1：未开始

| 状态名称 | 常见 ID | 说明 |
|---------|--------|------|
| 待确认 | 28 | 新建，等待确认 |
| 待处理 | 100005 | 已确认，等待分配 |
| 已确认 | 32 | 已确认并分配 |

### 阶段 2：进行中

| 状态名称 | 常见 ID | 说明 |
|---------|--------|------|
| 进行中 | 100010 | 正在处理 |
| 开发完成 | 100011 | 代码修复完成，待测试 |
| 测试中 | 100012 | 正在测试 |
| 待发布 | 100013 | 修复已验证，待部署 |

### 阶段 3：已关闭

| 状态名称 | 常见 ID | 说明 |
|---------|--------|------|
| 已修复 | 100020 | Bug 已修复并验证 |
| 已关闭 | 100021 | 已关闭（可能未修复） |
| 已拒绝 | 100022 | 不是 Bug 或不修复 |
| 重复 | 100023 | 与其他 Bug 重复 |

### 方法 2：按名称查找状态

```javascript
// 辅助函数：按名称查找状态 ID
function findStatusId(workflow, statusName) {
  const status = workflow.statuses.find(s => 
    s.name === statusName || 
    s.name.toLowerCase().includes(statusName.toLowerCase())
  )
  return status?.id
}

// 使用
const devCompleteId = findStatusId(workflow, "开发完成")
```

### 方法 3：按阶段查找状态

```javascript
// 获取特定阶段的所有状态
function getStatusesByStage(workflow, stage) {
  return workflow.statuses.filter(s => s.statusStage === stage.toString())
}

// 使用
const inProgressStatuses = getStatusesByStage(workflow, 2)
// 返回所有"进行中"阶段的状态
```

## 状态转换规则

### 有效转换

并非所有状态变更都被允许。使用工作流转换来确定有效的移动：

```javascript
// 检查转换是否有效
function canTransition(workflow, fromStatusId, toStatusId) {
  return workflow.transitions.some(t => 
    t.from === fromStatusId && t.to === toStatusId
  )
}

// 获取可用的下一个状态
function getNextStatuses(workflow, currentStatusId) {
  return workflow.transitions
    .filter(t => t.from === currentStatusId)
    .map(t => workflow.statuses.find(s => s.id === t.to))
}
```

### 常见转换路径

**典型 Bug 生命周期：**
```
待确认 (28) 
  → 已确认 (32)
  → 进行中 (100010)
  → 开发完成 (100011)
  → 测试中 (100012)
  → 已修复 (100020)
  → 已关闭 (100021)
```

**快速修复路径：**
```
待确认 (28)
  → 进行中 (100010)
  → 开发完成 (100011)
  → 已修复 (100020)
```

**拒绝路径：**
```
待确认 (28)
  → 已拒绝 (100022)
```

## 容错的状态更新

### 模式 1：先尝试后回退

```javascript
async function updateStatusSafely(organizationId, workItemId, targetStatusName) {
  // 先尝试常见 ID
  const commonIds = {
    "开发完成": "100011",
    "测试中": "100012",
    "已修复": "100020"
  }
  
  try {
    await mcp_aliyun_yunxiao_update_work_item({
      organizationId,
      workItemId,
      updateWorkItemFields: {
        status: commonIds[targetStatusName]
      }
    })
    return { success: true }
  } catch (error) {
    // 回退：查询工作流并重试
    const workflow = await mcp_aliyun_yunxiao_get_work_item_workflow({
      organizationId,
      workItemId
    })
    
    const correctId = findStatusId(workflow, targetStatusName)
    if (!correctId) {
      throw new Error(`未找到状态"${targetStatusName}"`)
    }
    
    await mcp_aliyun_yunxiao_update_work_item({
      organizationId,
      workItemId,
      updateWorkItemFields: {
        status: correctId
      }
    })
    
    return { success: true, usedFallback: true }
  }
}
```

### 模式 2：始终先查询（推荐）

```javascript
async function updateStatusReliably(organizationId, workItemId, targetStatusName) {
  // 始终先查询工作流
  const workflow = await mcp_aliyun_yunxiao_get_work_item_workflow({
    organizationId,
    workItemId
  })
  
  // 查找正确的状态 ID
  const statusId = findStatusId(workflow, targetStatusName)
  if (!statusId) {
    throw new Error(`未找到状态"${targetStatusName}"`)
  }
  
  // 验证转换是否有效
  const currentWorkItem = await mcp_aliyun_yunxiao_get_work_item({
    organizationId,
    workItemId
  })
  
  if (!canTransition(workflow, currentWorkItem.status, statusId)) {
    throw new Error(`无法从 ${currentWorkItem.status} 转换到 ${statusId}`)
  }
  
  // 更新状态
  await mcp_aliyun_yunxiao_update_work_item({
    organizationId,
    workItemId,
    updateWorkItemFields: {
      status: statusId
    }
  })
  
  return { success: true }
}
```

## 项目特定的自定义

某些云效项目可能有自定义状态。始终检查工作流：

### 示例：自定义状态名称

```javascript
// 项目 A 可能使用：
// "开发中" 而不是 "进行中"
// "待测试" 而不是 "测试中"

// 始终使用动态查找：
const workflow = await mcp_aliyun_yunxiao_get_work_item_workflow({
  organizationId,
  workItemId
})

// 灵活匹配
function findStatusFlexible(workflow, ...possibleNames) {
  for (const name of possibleNames) {
    const status = workflow.statuses.find(s => 
      s.name.includes(name) || name.includes(s.name)
    )
    if (status) return status
  }
  return null
}

// 使用
const inProgressStatus = findStatusFlexible(
  workflow, 
  "进行中", 
  "开发中", 
  "处理中"
)
```

## 最佳实践

### ✅ 应该做的

1. **始终动态查询工作流**用于生产环境
2. **缓存工作流数据**在单个 Bug 修复会话期间
3. **验证转换**在尝试状态更新前
4. **使用状态阶段过滤**减少搜索结果
5. **优雅处理缺失状态**提供清晰的错误消息

### ❌ 不应该做的

1. **不要硬编码状态 ID**在生产代码中
2. **不要假设状态名称**在不同项目间一致
3. **不要跳过转换验证** - 无效转换会失败
4. **不要在更新状态前不读取当前状态**
5. **不要忘记在更改状态时添加评论**

## 故障排查

### 问题：状态更新失败

**症状：**
```
错误：无效的状态 ID
```

**解决方案：**
1. 查询工作流获取有效的状态 ID
2. 验证状态在项目中存在
3. 检查从当前状态的转换是否被允许

### 问题：找不到状态

**症状：**
```
未找到状态"开发完成"
```

**解决方案：**
1. 检查中文 vs 英文名称不匹配
2. 使用灵活匹配（部分字符串匹配）
3. 列出所有可用状态让用户选择

### 问题：不允许转换

**症状：**
```
无法从状态 A 转换到状态 B
```

**解决方案：**
1. 检查工作流转换
2. 如果不允许直接转换，查找中间状态
3. 如有必要，通过多个步骤更新

## 参考：完整状态生命周期

```
待确认 → 已确认 → 进行中 → 开发完成 → 测试中 → 已修复 → 已关闭
   ↓        ↓                                    ↑
已拒绝    重复 ←──────────────────────────────────┘
```

## 总结

- **状态 ID 因项目而异** - 始终动态查询
- **使用 statusStage** 进行广泛过滤（1=未开始，2=进行中，3=已关闭）
- **在更新状态前验证转换**
- **实现容错模式**以实现健壮的状态更新
- **在会话期间缓存工作流数据**以减少 API 调用
